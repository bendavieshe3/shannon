### Gotchas

- **Schema codification structural rather than substring.** AC#4 names the four-category schema as `Drift / Gap / Internal contradiction / Strength`. My `alignment.md` codifies the schema by giving each category its own bulleted definition with bold-label form (`**Drift**`, `**Gap**`, etc.). The combined slash-separated phrase does not appear as a single substring; the four labels are individually plain-prose-grep-verifiable (the surrounding `**` markers are outside the substring per parent Epic AC#7 discipline). This is semantically equivalent to the AC's intent — the schema is codified — but a Gate-3 reviewer expecting to grep the combined phrase as a single substring would not find it. Surfaced for Gate 3 review.
- **`allowed-tools:` field name follows AC#2 verbatim.** The canonical YAML frontmatter field per AC#2 R-2 is `allowed-tools:` (with the colon-then-list form). If the Claude Code runtime expects the more terse `tools:` field name instead — which is what some standard Claude Code subagent docs name — the tool restriction codified at line 4 may not take effect at runtime. The first lived test of whether `allowed-tools:` is honoured will be the TASK-018 dogfood run; sibling Tasks TASK-013 / TASK-014 will inherit the same field name and either confirm-or-refute the convention. This is the held G-1 routing's first lived datapoint.
- **`./.claude/` gitignored architecture confirmed again.** `alignment.md` is filesystem-present at `./.claude/skills/shannon-supervisor/checkers/alignment.md` (verified by `ls -la`) but not Git-tracked (`git status` shows only the lifecycle-bookkeeping modifications). Same architecture as TASK-011's `SKILL.md`. The shipping-source mirror deferral per AC#6 remains intact — the file lives only in the deployed copy, not in any versioned shipping source until the future `/shannon-setup`-update Epic creates `./shannon/skills/shannon-supervisor/checkers/alignment.md`.
