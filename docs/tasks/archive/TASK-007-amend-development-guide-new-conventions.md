### Approach

A single-file editing pass against `development_guide.md` landing six distinct additions plus an in-place metadata refresh, followed by additivity verification, anchor verification, and scope-confined-diff verification. **Edit order — bottom-to-top, because every substantive insertion adds lines.** AC1 (~1–3 lines at line 78), AC3 (~10 lines between lines 145 and 147), AC4 (~1 line after line 112), AC5 (~5–8 lines above line 179), and AC6 (~1 line within lines 138–145) all *increase* the file's line count at their insertion points. Top-to-bottom would invalidate every pinned line number below the first edit. **Bottom-to-top makes each insertion happen below every still-pending insertion point**, so the line anchors stated in Requirements remain valid until each anchor's own edit lands. AC9 (the top-of-file metadata refresh at lines 3–5) is an in-place word-level edit, line-count-invariant, so it can run at any point — sequenced last so all preceding line shifts are already factored. This is the line-direction principle TASK-004's Plan captured the converse of: word-level in-line replacements are line-count-invariant and may be worked top-to-bottom as a reading-order discipline; line-count-changing edits must be worked bottom-to-top so each insertion sits below every pending one.
